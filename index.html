Day 1 — O que é o Kubernetes?

O ecossistema de containers: Engine, OCI e Runtime

Antes de entrar no Kubernetes, é essencial entender o que está embaixo dele. Existe uma cadeia de componentes que trabalham juntos para que um container seja executado, e confundir esses papéis é um erro comum mesmo entre pessoas experientes.

O Container Engine

O Container Engine é a camada mais alta da hierarquia. Ele é responsável por todo o gerenciamento de alto nível: baixar imagens do registry, gerenciar volumes, configurar rede, gerenciar o ciclo de vida dos containers e oferecer uma interface amigável ao usuário (como a CLI do Docker). Quando você digita docker run nginx, é o Container Engine que interpreta esse comando e orquestra tudo.

As opções mais importantes hoje são o Docker (o mais popular, usa containerd internamente), o Podman (alternativa sem daemon, rootless por padrão, compatível com Docker CLI) e o CRI-O (criado especificamente para Kubernetes, implementa a CRI nativa).

A OCI — Open Container Initiative

Em 2015, as maiores empresas do setor — Docker, CoreOS, Google, IBM, Microsoft, Red Hat e VMware — perceberam que os containers precisavam de um padrão aberto. Caso contrário, cada fornecedor criaria seu próprio formato incompatível. Nasceu então a OCI, hoje parte da Linux Foundation.

A OCI define dois padrões cruciais: o Image Spec (como uma imagem de container deve ser estruturada) e o Runtime Spec (como executá-la). O projeto mais importante gerado pela OCI é o runc, um runtime de baixo nível escrito em Go, que é usado por praticamente todos os Container Engines do mercado.

💡

Graças à OCI, uma imagem construída com Docker pode ser executada por containerd, CRI-O ou qualquer runtime compatível. Isso é o que permite ao Kubernetes ser agnóstico ao runtime.

O Container Runtime — a linha de frente

O Container Runtime é quem faz o trabalho pesado de fato: ele conversa diretamente com o kernel Linux para criar os namespaces, configurar os cgroups, montar os sistemas de arquivos e executar o processo do container.

Existem três categorias:

  • Low-level (nível do kernel): o runc, crun e runsc. Eles recebem um bundle de container no formato OCI e o executam diretamente via syscalls do Linux.
  • High-level (usado pelos Engines): o containerd (padrão do Docker e do K8s) e o CRI-O. Eles gerenciam imagens, snapshots e chamam o runtime low-level.
  • Sandbox e Virtualized: para máxima segurança. O gVisor (Google) usa um micro-kernel para interceptar syscalls, e o Kata Containers executa cada container em uma VM leve.
🚨

Atenção, Docker no K8s 1.24+: o Docker não implementa a interface CRI (Container Runtime Interface) do Kubernetes. A partir da versão 1.24, o suporte foi removido. Use containerd ou CRI-O em clusters modernos. O Docker ainda é ótimo para desenvolvimento local, mas não é o runtime do K8s em produção.

O que é o Kubernetes e por que ele existe?

Imagine que você tem 100 contêineres rodando em 20 servidores. Um container travou às 3h da manhã. Como você sabe? Como ele volta? Qual servidor tem recursos livres para receber um novo container? Como você garante que 5 réplicas estejam sempre rodando? Como você atualiza a versão da aplicação sem derrubar o serviço? Gerenciar isso manualmente é impossível em escala. Essa é exatamente a dor que o Kubernetes resolve.

O Kubernetes (k8s) é uma plataforma open-source de orquestração de containers, desenvolvida pelo Google em 2014 e doada à CNCF (Cloud Native Computing Foundation). O nome vem do grego: "timoneiro" — quem controla o leme de um navio. A comunidade o apelidou de k8s (k + 8 letras + s), pronunciado "kates".

A história: Borg → Omega → Kubernetes

O Google gerencia containers em larga escala há mais de 15 anos. Seu primeiro sistema foi o Borg, criado para gerenciar serviços de longa duração e jobs em lote em escala planetária. O Borg continua sendo o principal sistema interno do Google. Depois veio o Omega, construído do zero com uma arquitetura mais consistente. O Kubernetes foi o terceiro sistema — desta vez open source, pensado para a comunidade e para o mercado de cloud.

ℹ️

Por que "k8s" importa na prática? O Kubernetes resolve três problemas fundamentais de produção: alta disponibilidade (self-healing automático), escalabilidade (scaling horizontal declarativo) e gerenciamento de releases (rolling updates e rollbacks sem downtime).

O que o Kubernetes faz por você

O K8s opera no modelo declarativo: você descreve o que quer (estado desejado) em um arquivo YAML, e o Kubernetes garante que o mundo real corresponda a esse estado — para sempre. Se um container morrer, ele recria. Se um node falhar, redistribui. Se a carga aumentar, escala.

Arquitetura do Kubernetes: Control Plane e Workers

O Kubernetes segue um modelo control plane / workers. O control plane é o "cérebro" — ele toma decisões, armazena estado e gerencia o cluster. Os workers são os "músculos" — eles executam as cargas de trabalho (seus containers). Em produção, você tem 3 ou 5 nodes de control plane (para HA) e N workers.

⚠️

É possível criar um cluster com apenas 1 node (control plane + worker no mesmo host). Isso é útil para estudos com Minikube ou Kind, mas nunca para produção. Se o único node cair, o cluster todo cai.

Control Plane: os 5 componentes

1. kube-apiserver — a porta de entrada
O kube-apiserver é o único ponto de entrada de todo o cluster. Cada comando kubectl, cada ação de qualquer componente interno, passa obrigatoriamente pelo apiserver via REST/gRPC. Ele autentica, autoriza, valida e persiste tudo. Se o apiserver estiver fora, o cluster para de aceitar novas requisições (mas cargas já rodando continuam — o K8s é resiliente).

2. etcd — a memória do cluster
O etcd é um banco de dados chave-valor distribuído que usa o protocolo Raft para consenso. Todo o estado do cluster — cada Pod, cada Deployment, cada Secret, cada ConfigMap — está armazenado no etcd. Se você perder o etcd sem backup, você perde o cluster inteiro. Por isso, em produção: backup diário do etcd é obrigatório. O etcd roda exclusivamente nos nodes de control plane por segurança.

3. kube-scheduler — o alocador de recursos
Quando um Pod é criado sem um node atribuído, o scheduler entra em ação. Ele analisa dois critérios em sequência: primeiro faz a filtragem (elimina nodes que não têm recursos suficientes, têm taints incompatíveis, etc.) e depois o scoring (pontua os nodes elegíveis e escolhe o melhor). O scheduler registra a decisão no etcd via apiserver, e o kubelet do node escolhido recebe a instrução.

4. kube-controller-manager — o reconciliador
Este é o coração do self-healing. O controller-manager roda dezenas de loops de controle (reconciliation loops) que ficam constantemente comparando o "estado desejado" com o "estado real". Se divergirem, ele age: o Replication Controller recria pods mortos, o Node Controller detecta nodes caídos, o Endpoints Controller atualiza os IPs dos Services. Cada controller tem uma única responsabilidade.

5. cloud-controller-manager — a ponte com a cloud
Em ambientes de cloud (AWS, GCP, Azure), este componente é responsável por provisionar recursos de infraestrutura: criar Load Balancers quando você cria um Service do tipo LoadBalancer, provisionar discos EBS quando você cria um PVC, etc.

Worker Nodes: os 3 componentes

kubelet — o agente no node
O kubelet é o agente que roda em cada worker node (e também no control plane). Ele é responsável por receber as especificações de Pod do apiserver e garantir que os containers descritos estejam rodando. O kubelet fala com o container runtime via CRI (Container Runtime Interface), seja containerd ou CRI-O.

kube-proxy — o guardião da rede
O kube-proxy mantém as regras de rede em cada node usando iptables ou IPVS. Quando você cria um Service, o kube-proxy garante que o tráfego chegue nos Pods certos, em qualquer node do cluster. É ele quem viabiliza o load balancing interno do Kubernetes.

Container Runtime — o executor
O runtime (containerd, CRI-O) recebe as instruções do kubelet e executa efetivamente os containers, gerenciando imagens, snapshots e a execução dos processos.

Instalando e dominando o kubectl

O kubectl é a CLI oficial do Kubernetes. Tudo que você faz no cluster — criar recursos, verificar status, ver logs, fazer rollbacks — passa pelo kubectl, que traduz seus comandos em requisições REST para o kube-apiserver.

BASH · INSTALAÇÃO
# Linux: baixa o binário da versão estável mais recente
curl -LO "https://dl.k8s.io/release/$(curl -L -s https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl"
sudo install -o root -g root -m 0755 kubectl /usr/local/bin/kubectl
kubectl version --client

# macOS via Homebrew
sudo brew install kubectl
kubectl version --client

Configurando autocomplete e alias — economize horas

Estes dois passos são obrigatórios antes de qualquer estudo ou prova de certificação:

BASH · PRODUTIVIDADE
# Autocomplete no Bash (adiciona permanentemente ao ~/.bashrc)
source <(kubectl completion bash)
echo "source <(kubectl completion bash)" >> ~/.bashrc

# Autocomplete no Zsh
source <(kubectl completion zsh)
echo "[[ \$commands[kubectl] ]] && source <(kubectl completion zsh)" >> ~/.zshrc

# Alias 'k' — economiza centenas de caracteres por dia
alias k=kubectl
complete -F __start_kubectl k
# Agora 'k get pods' funciona igual a 'kubectl get pods'

Os comandos mais usados no dia a dia

Aprender kubectl é como aprender um novo idioma — você usa os mesmos padrões repetidamente até ficarem automáticos.

O --dry-run=client -o yaml é a dica de ouro das provas CKA e CKAD. Em vez de escrever o YAML do zero, você gera um template e edita apenas o que precisa. Economiza tempo e evita erros de sintaxe.

Criando seu primeiro cluster Kubernetes local

Para estudar Kubernetes, você precisa de um cluster para experimentar. Felizmente, existem ferramentas que criam clusters completos na sua máquina em questão de minutos.

Kind — Kubernetes in Docker (recomendado para estudos)

O Kind cria nodes de Kubernetes como containers Docker. A grande vantagem é que você pode criar um cluster com múltiplos nodes (control-plane + workers) em uma única máquina, simulando um ambiente mais próximo da realidade. É também muito usado em pipelines de CI/CD.

BASH · KIND
# Instalação no Linux
curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.20.0/kind-linux-amd64
chmod +x ./kind && sudo mv ./kind /usr/local/bin/kind

# Criar cluster simples (1 node)
kind create cluster

# Criar cluster com múltiplos nodes (mais realista)
cat <<EOF | kind create cluster --config=-
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
- role: worker
- role: worker
EOF

# Verificar os nodes
kubectl get nodes
# NAME                 STATUS   ROLES           AGE
# kind-control-plane   Ready    control-plane   2m
# kind-worker          Ready    <none>          90s
# kind-worker2         Ready    <none>          90s

# Verificar os pods do sistema (CoreDNS, kube-proxy, etc.)
kubectl get pods -n kube-system

# Remover o cluster
kind delete cluster

Minikube — para quem quer simplicidade

O Minikube cria um cluster com um único node que funciona tanto como control plane quanto worker. É mais simples que o Kind, mas tem menos capacidade de simular ambientes multi-node. Requer: 2 CPUs, 2GB RAM e 20GB de HD livre.

BASH · MINIKUBE
# Instalação no Linux
curl -Lo minikube https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
chmod +x ./minikube && sudo mv ./minikube /usr/local/bin/minikube

# Iniciar o cluster
minikube start

# Comandos úteis do Minikube
minikube status           # verifica status
minikube ip               # endereço IP do cluster
minikube ssh              # acessa o node via SSH
minikube dashboard        # abre dashboard web
minikube service meu-svc  # abre service no browser
minikube stop             # para o cluster
minikube delete           # remove completamente

# Primeiros passos: deploy e exposição

O Pod: a menor unidade do Kubernetes

Se você veio do Docker, está acostumado a pensar em termos de containers. No Kubernetes, você precisa mudar esse modelo mental: o K8s não gerencia containers diretamente — ele gerencia Pods. Um Pod é um wrapper em torno de um ou mais containers que compartilham o mesmo contexto de execução.

Pense no Pod como um "host virtual" dentro do cluster. Todos os containers dentro do mesmo Pod enxergam o mesmo endereço IP, a mesma tabela de rotas e os mesmos ports. Se o container A está escutando na porta 8080 e o container B quer falar com ele, basta usar localhost:8080. Eles são vizinhos no mesmo "apartamento".

Características fundamentais dos Pods

  • Compartilham a mesma rede: mesmo IP, mesmos ports, comunicação via localhost entre containers do mesmo Pod.
  • Podem compartilhar volumes: se você montar um volume no Pod, todos os containers têm acesso — ideal para o padrão sidecar.
  • São efêmeros por natureza: quando um Pod morre, ele não ressuscita. Um novo Pod é criado no lugar, com um novo IP e novo nome. Por isso, nunca gerencie Pods diretamente em produção.
  • A unidade de scheduling: o kube-scheduler aloca Pods em Nodes, não containers individuais. Todos os containers de um Pod vivem no mesmo Node.

Por que não usar containers diretamente?

O Kubernetes precisa de uma abstração acima do container porque alguns casos requerem que múltiplos processos coexistam com contexto compartilhado. O padrão sidecar é o exemplo clássico: imagine uma aplicação que gera logs em arquivo. Você pode ter o container principal da aplicação e um container sidecar de log shipper (como o Fluentd) que lê esse arquivo e o envia para o Elasticsearch — ambos no mesmo Pod, compartilhando um volume.

ℹ️

Regra de ouro: só coloque múltiplos containers no mesmo Pod quando eles têm um acoplamento muito forte — quando precisam compartilhar recursos locais e não faz sentido existirem separados. Para containers independentes, use Pods separados.

Criando, inspecionando e gerenciando Pods

Existem duas formas de criar um Pod: o modo imperativo (via linha de comando) e o modo declarativo (via arquivo YAML). O modo imperativo é rápido para testes e é usado nas provas de certificação. O modo declarativo é o correto para produção, pois o manifesto pode ser versionado no Git.

BASH · IMPERATIVO
# Criar um Pod diretamente (bom para testes rápidos)
kubectl run giropops --image=nginx --port=80

# Ver o Pod criado — diferentes formatos de saída
kubectl get pods
kubectl get pods -o wide        # mostra IP e Node — use sempre!
kubectl get pods -o yaml        # YAML completo (revela campos que você não especificou)
kubectl get pods -n kube-system # pods do sistema
kubectl get pods -A             # todos os namespaces

# Inspecionar detalhes — EVENTS é o mais importante para debug
kubectl describe pods giropops

# Ver logs
kubectl logs giropops            # logs até o momento
kubectl logs -f giropops         # follow: streaming em tempo real

# Remover o Pod
kubectl delete pods giropops

Criando via YAML — a forma profissional

O arquivo YAML é o coração do Kubernetes. Cada campo tem um significado preciso. Vamos destrinchar linha a linha:

YAML · pod.yaml
apiVersion: v1           # versão da API — para Pods é sempre v1
kind: Pod               # tipo do recurso
metadata:              # metadados: quem você é
  name: giropops       # nome único dentro do namespace
  labels:              # rótulos chave/valor para seleção
    run: giropops      # Services usam labels para selecionar pods!
    env: producao
spec:                  # especificação: o que você quer
  containers:          # lista de containers (pode ter mais de um)
  - name: giropops     # nome do container dentro do Pod
    image: nginx:1.25   # imagem com tag específica (nunca use :latest em produção!)
    ports:
    - containerPort: 80 # documentação: a porta não cria regra de firewall
BASH · APLICANDO
# apply: cria OU atualiza — idempotente (pode executar múltiplas vezes)
kubectl apply -f pod.yaml

# Diferença entre apply e create:
# apply: declarativo, atualiza se já existe, registra anotação de configuração
# create: imperativo, falha se já existe, mais simples mas menos flexível

# Dry-run: valida sem aplicar
kubectl apply -f pod.yaml --dry-run=client

# Ver o que mudaria sem aplicar
kubectl diff -f pod.yaml
💡

Use kubectl explain pod.spec.containers para ver a documentação de qualquer campo do manifesto diretamente no terminal. Extremamente útil durante provas de certificação onde você não pode usar outros recursos além de kubernetes.io.

Pod com múltiplos containers: attach vs exec

Quando você tem dois containers no mesmo Pod, precisa saber como interagir com cada um. O Kubernetes oferece dois comandos: attach e exec, que têm comportamentos completamente diferentes — e confundi-los pode causar problemas.

O comando kubectl exec cria um novo processo dentro do container (geralmente um shell como sh ou bash). É o equivalente ao docker exec -it. Você tem total controle — pode executar qualquer comando, listar arquivos, inspecionar variáveis de ambiente, testar conectividade.

O comando kubectl attach conecta ao processo principal já em execução no container. É como conectar o seu terminal ao stdin/stdout do processo em foreground. Se você usar attach em um container Nginx, você ficará "travado" olhando para os logs do Nginx sem conseguir digitar nada útil. Use attach apenas em containers que têm um processo interativo como principal (terminal REPL, por exemplo).

YAML · pod-multi-container.yaml
apiVersion: v1
kind: Pod
metadata:
  name: giropops
  labels:
    run: giropops
spec:
  containers:
  - name: girus          # container principal: Nginx
    image: nginx         # tem processo em foreground (o servidor web)
    ports:
    - containerPort: 80
  - name: strigus        # container auxiliar: Alpine
    image: alpine        # Alpine não tem processo em foreground!
    args:
    - sleep
    - "infinity"         # sem isso o container morre imediatamente
BASH · INTERAGINDO COM CONTAINERS
kubectl apply -f pod-multi-container.yaml
kubectl get pods  # READY: 2/2 indica 2 containers rodando

# exec no Alpine — cria um shell novo, interativo
kubectl exec giropops -c strigus -it -- sh
# Dentro do container: 'hostname', 'ip addr', 'wget -O- localhost:80' (fala com Nginx!)

# exec no Nginx — também funciona com exec
kubectl exec giropops -c girus -it -- sh

# exec sem interatividade: executar um único comando
kubectl exec giropops -c strigus -- ls /etc
kubectl exec giropops -c strigus -- env

# attach no Alpine (tem o sleep como processo principal)
kubectl attach giropops -c strigus

Limites de CPU e Memória: requests e limits

Um dos erros mais graves que se pode cometer em Kubernetes é não definir limites de recursos. Sem limites, um único container mal comportado pode consumir toda a memória de um Node e derrubar todos os outros Pods que rodam nele. Isso é conhecido como noisy neighbor problem.

O Kubernetes tem dois campos para controlar recursos: requests e limits. Entender a diferença entre eles é fundamental.

requests — a garantia mínima

O requests é o que o scheduler usa para decidir onde colocar o Pod. Quando o scheduler analisa os Nodes disponíveis, ele só considera os Nodes que têm os recursos de requests disponíveis. O container tem esses recursos garantidos. Se um Node tem 4GB de RAM e você tem 10 Pods com memory requests: 500Mi, o Node pode acomodar todos (10 × 500Mi = 5GB, mas os Pods podem usar menos que o solicitado).

limits — o teto absoluto

O limits é o máximo absoluto que o container pode consumir. Ultrapassar o limite de memória resulta em OOMKilled — o kernel Linux mata o processo. Ultrapassar o limite de CPU não mata o container, apenas throttle (desacelera) o processo. Em produção, defina sempre os dois.

Unidades de medida

Para memória: o Kubernetes usa o sistema binário. Mi = mebibytes (1 MiB = 1.048.576 bytes), Gi = gibibytes. Evite usar M e G (sistema decimal) para não ter surpresas. Para CPU: 1 CPU = 1000m (millicores). 0.5 e 500m são equivalentes — representam metade de um core.

YAML · pod-limitado.yaml
apiVersion: v1
kind: Pod
metadata:
  name: giropops
spec:
  containers:
  - name: girus
    image: nginx
    ports:
    - containerPort: 80
    resources:
      limits:            # TETO: exceder memória = OOMKilled; exceder CPU = throttle
        memory: "128Mi"  # 128 mebibytes máximo
        cpu: "0.5"        # 500 millicores máximo (metade de um core)
      requests:          # GARANTIA: o scheduler usa isso para alocar no Node
        memory: "64Mi"   # 64Mi garantidos
        cpu: "0.3"        # 300m garantidos (30% de um core)

Volume EmptyDir: compartilhando dados entre containers

Um volume do tipo EmptyDir é criado quando o Pod é criado e destruído quando o Pod é destruído. Ele é um volume efêmero — dados não sobrevivem ao Pod. Isso pode parecer pouco útil à primeira vista, mas o EmptyDir resolve um problema específico e comum: compartilhar dados entre os containers do mesmo Pod.

Imagine o padrão sidecar de log: o container principal da aplicação grava logs em /var/log/app. O container sidecar de log shipper precisa ler esses arquivos e enviá-los para o Elasticsearch. Como? Ambos montam o mesmo EmptyDir no mesmo path. Os dados fluem entre eles através do volume compartilhado, dentro do mesmo Pod.

O EmptyDir também é útil para cache temporário de alta velocidade, especialmente com medium: Memory que usa RAM em vez de disco — dramaticamente mais rápido.

YAML · pod-emptydir.yaml
apiVersion: v1
kind: Pod
metadata:
  name: giropops
spec:
  containers:
  - name: girus
    image: ubuntu
    args: ["sleep", "infinity"]
    volumeMounts:                    # onde o volume será montado NESTE container
    - name: primeiro-emptydir        # referência ao volume definido abaixo
      mountPath: /giropops           # caminho dentro do container
  volumes:                           # volumes disponíveis para o Pod
  - name: primeiro-emptydir         # nome para referenciar no volumeMounts
    emptyDir:                        # tipo: diretório vazio temporário
      sizeLimit: 256Mi               # limite de tamanho do diretório
      # medium: Memory  ← descomente para usar RAM (tmpfs) em vez de disco!

O que é um Deployment e por que ele existe?

Quando estamos criando um Deployment no Kubernetes, automaticamente estamos criando além do próprio Deployment um ReplicaSet, e esse ReplicaSet é quem vai criar e gerenciar os Pods que estão dentro do Deployment. É uma hierarquia de três camadas: Deployment → ReplicaSet → Pods.

Por que essa estrutura? O Pod sozinho é efêmero e sem auto-recuperação. O ReplicaSet garante que N réplicas estejam sempre rodando, mas não sabe como fazer uma atualização de versão de forma controlada. O Deployment resolve exatamente isso: ele gerencia os ReplicaSets ao longo do tempo, criando novos ReplicaSets para novas versões e mantendo os antigos para possibilitar rollbacks.

Quando você atualiza a imagem de um Deployment, o que acontece nos bastidores é fascinante: o Deployment cria um novo ReplicaSet com a nova versão, e gradualmente aumenta as réplicas do novo enquanto diminui as do antigo, até que o antigo tenha 0 réplicas — mas ele não é removido. Ele fica lá, zerado, esperando. Se você precisar de um rollback, o Deployment simplesmente inverte o processo, reabastecendo o ReplicaSet antigo.

💡

A hierarquia completa em uma linha: Deployment gerencia ReplicaSets → ReplicaSet gerencia Pods → Pod contém Containers. Cada camada tem sua responsabilidade específica e não deve ser pulada.

Criando e gerenciando um Deployment

O manifesto de um Deployment é mais extenso que o de um Pod, mas segue a mesma lógica. O ponto mais crítico e que causa mais confusão é o relacionamento entre o campo spec.selector.matchLabels e o spec.template.metadata.labels: eles obrigatoriamente devem ser iguais. É assim que o Deployment identifica quais Pods ele gerencia.

YAML · nginx-deployment.yaml
apiVersion: apps/v1             # Deployments usam apps/v1, não v1
kind: Deployment
metadata:
  labels:
    app: nginx-deployment
  name: nginx-deployment
spec:
  replicas: 3                   # quantas réplicas do Pod manter sempre
  selector:
    matchLabels:
      app: nginx-deployment    # ← DEVE bater exatamente com template.metadata.labels
  strategy: {}                  # {} = usa RollingUpdate com valores padrão
  template:                     # template dos Pods que o ReplicaSet vai criar
    metadata:
      labels:
        app: nginx-deployment  # ← DEVE bater com selector.matchLabels acima
    spec:
      containers:
      - image: nginx
        name: nginx
        resources:
          limits:
            cpu: "0.5"
            memory: 256Mi
          requests:
            cpu: "0.25"
            memory: 128Mi

Estratégia RollingUpdate — zero downtime

A estratégia padrão do Kubernetes para atualizar Deployments é a RollingUpdate. Em vez de derrubar tudo e subir a versão nova de uma vez, o K8s faz isso gradualmente: sobe novos Pods com a versão nova, espera ficarem prontos e só então derruba os Pods com a versão velha. O serviço nunca fica totalmente indisponível.

Dois parâmetros controlam o comportamento do rolling update:

  • maxSurge: define quantos Pods a mais (acima do número de réplicas desejado) o Deployment pode criar temporariamente durante a atualização. Se você tem 10 réplicas e maxSurge: 1, o K8s pode ter temporariamente 11 Pods rodando.
  • maxUnavailable: define quantos Pods podem ficar indisponíveis durante a atualização. Com maxUnavailable: 0, você garante que sempre haverá o número exato de réplicas disponíveis — zero downtime absoluto.

O processo de uma atualização com maxSurge: 1 e maxUnavailable: 0 em um Deployment com 3 réplicas funciona assim: o K8s cria 1 Pod novo (agora tem 4), espera ele ficar Ready, então mata 1 Pod antigo (volta para 3), cria outro Pod novo (4 novamente), espera ficar Ready, mata outro antigo... e assim por diante até todos os 3 serem da nova versão.

YAML · rolling update configurado
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1           # pode criar 1 Pod extra durante o update
      maxUnavailable: 2   # pode ter 2 Pods indisponíveis simultaneamente
  template:
    spec:
      containers:
      - image: nginx:1.16.0  # mude aqui para disparar o rolling update
        name: nginx
BASH · ACOMPANHANDO O ROLLOUT
kubectl apply -f nginx-deployment.yaml

# Acompanhar o rolling update em tempo real
kubectl rollout status deployment nginx-deployment
# Waiting for deployment "nginx-deployment" rollout to finish: 5 out of 10 new replicas...
# deployment "nginx-deployment" successfully rolled out

# Ver os dois ReplicaSets: novo (cheio) e velho (zerado, mas preservado)
kubectl get replicasets
# NAME                          DESIRED   CURRENT   READY   AGE
# nginx-deployment-6dd8d7cfbd   0         0         0       15m  ← versão antiga
# nginx-deployment-7858bcf56f   10        10        10      2m   ← versão nova

# Confirmar qual versão está rodando nos Pods
kubectl exec nginx-deployment-7858bcf56f-xxxxx -- nginx -v

Estratégia Recreate — quando o downtime é aceitável

A estratégia Recreate é simples e brutal: o Kubernetes derruba todos os Pods existentes antes de criar os novos. Haverá downtime entre o momento em que o último Pod antigo morrer e o primeiro Pod novo ficar pronto.

Quando usar o Recreate? Em casos em que sua aplicação não tolera duas versões rodando em paralelo. O exemplo clássico é uma migração de schema de banco de dados: se a versão nova faz alterações no schema que são incompatíveis com a versão antiga, você não pode ter ambas rodando simultaneamente. Outra situação: aplicações com licença por instância que não permitem múltiplas versões ativas.

Para a maioria das aplicações modernas (APIs stateless, microsserviços), o RollingUpdate é sempre preferível. Reserve o Recreate para casos com restrições específicas de compatibilidade.

YAML · recreate strategy
spec:
  replicas: 10
  strategy:
    type: Recreate   # sem subparâmetros — tudo desce, tudo sobe
  template:
    spec:
      containers:
      - image: nginx:1.19.0
        name: nginx
BASH · OBSERVANDO O RECREATE
kubectl apply -f nginx-deployment.yaml

# Acompanhar em outra janela enquanto faz o apply
kubectl get pods -w
# TODOS os pods antigos terminam (Terminating) antes dos novos começarem
# Esse é o intervalo de downtime!

kubectl rollout status deployment nginx-deployment

Rollback — voltando no tempo de forma controlada

Um dos recursos mais valiosos do Deployment é o histórico de revisões. Toda vez que você atualiza a imagem ou qualquer campo do spec.template, o K8s salva a revisão anterior. Você pode voltar para qualquer revisão salva com um único comando.

O número de revisões mantidas é controlado pelo campo spec.revisionHistoryLimit, que tem valor padrão 10. Em clusters grandes com muitos Deployments, considere reduzir para 3-5 para economizar recursos no etcd.

Uma dica muito útil para produção: adicione uma annotation de kubernetes.io/change-cause em cada atualização. Assim o histórico de revisões fica legível com mensagens descritivas, como um git log.

BASH · HISTÓRICO E ROLLBACK
# Ver o histórico de revisões
kubectl rollout history deployment nginx-deployment
# REVISION  CHANGE-CAUSE
# 1         <none>
# 2         <none>
# 3         <none>

# Ver o que havia em uma revisão específica
kubectl rollout history deployment nginx-deployment --revision=1
# Mostra a imagem e toda a spec do template dessa revisão

# Rollback para a revisão imediatamente anterior
kubectl rollout undo deployment nginx-deployment

# Rollback para uma revisão específica
kubectl rollout undo deployment nginx-deployment --to-revision=1

# Confirmar que voltou
kubectl rollout status deployment nginx-deployment
kubectl get replicasets  # o ReplicaSet antigo voltou a ter réplicas

# ─── ADICIONANDO CHANGE-CAUSE AO HISTÓRICO ───────────────────────
# Após cada apply, anote a causa:
kubectl annotate deployment nginx-deployment \
  kubernetes.io/change-cause="Atualizando para nginx 1.19.2 - fix CVE-2021-XXXX"

# Agora o histórico é legível:
kubectl rollout history deployment nginx-deployment
# REVISION  CHANGE-CAUSE
# 1         Deploy inicial nginx 1.16.0
# 2         Atualizando para nginx 1.19.0
# 3         Atualizando para nginx 1.19.2 - fix CVE-2021-XXXX

# Controlar quantas revisões manter
# Adicione no spec do Deployment:
# spec.revisionHistoryLimit: 5  (padrão é 10)
✅

O rollback é instantâneo — o K8s simplesmente redistribui as réplicas entre os ReplicaSets existentes. Em alguns segundos, você está de volta para a versão anterior. Isso é possível porque o ReplicaSet antigo nunca foi deletado, apenas zerado.

Day 4 — ReplicaSets, DaemonSets e Probes

ReplicaSet: entendendo o mecanismo por trás dos Deployments

O ReplicaSet é o objeto que garante que um determinado número de réplicas de um Pod esteja em execução a qualquer momento. Se um Pod morrer por qualquer razão — o processo crashou, o Node foi reiniciado, você deletou acidentalmente — o ReplicaSet detecta que o número de réplicas está abaixo do desejado e imediatamente cria um novo Pod para compensar.

Na prática, você raramente cria ReplicaSets diretamente. O Deployment faz isso por você. Mas entender como o ReplicaSet funciona é essencial para compreender comportamentos como o rollback (o Deployment simplesmente muda qual ReplicaSet está ativo).

O problema de atualizar um ReplicaSet diretamente

Aqui está algo que ilustra perfeitamente por que você deve usar Deployments em vez de ReplicaSets: se você criar um ReplicaSet com a imagem nginx:1.19.2 e depois atualizar o manifesto para nginx:1.19.3 e fazer kubectl apply, o ReplicaSet vai aceitar a atualização — mas os Pods existentes não serão recriados. Apenas Pods novos (criados após a atualização) usarão a nova imagem. O resultado é um cluster com duas versões do mesmo serviço rodando, sem que você tenha pedido isso explicitamente.

O Deployment resolve esse problema porque ao detectar uma mudança no template, ele cria um novo ReplicaSet e gerencia a transição de forma controlada.

YAML · nginx-replicaset.yaml
apiVersion: apps/v1
kind: ReplicaSet           # mesma apiVersion do Deployment
metadata:
  labels:
    app: nginx-app
  name: nginx-replicaset
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx-app          # seleciona Pods com este label
  template:
    metadata:
      labels:
        app: nginx-app        # deve bater com selector
    spec:
      containers:
      - image: nginx:1.19.2
        name: nginx
        resources:
          limits: {cpu: "0.5", memory: 256Mi}
          requests: {cpu: "0.25", memory: 128Mi}
BASH · OBSERVANDO O PROBLEMA DO REPLICASET
kubectl apply -f nginx-replicaset.yaml
kubectl get replicasets

# Agora mude a imagem para 1.19.3 no YAML e aplique novamente
kubectl apply -f nginx-replicaset.yaml  # aplica sem erro

# Os Pods NÃO foram atualizados — ainda usam 1.19.2!
kubectl get pods -o=jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[0].image}{"\n"}{end}'
# nginx-replicaset-xxx   nginx:1.19.2  ← versão antiga ainda aqui

# Só se você deletar um Pod manualmente, o novo virá com 1.19.3
kubectl delete pod nginx-replicaset-XXXXX
# O novo Pod criado usará 1.19.3, mas os outros dois ainda usam 1.19.2!
# ← PROBLEMA: duas versões rodando ao mesmo tempo sem controle

# Remover o ReplicaSet (e seus Pods)
kubectl delete replicaset nginx-replicaset
# ou: kubectl delete -f nginx-replicaset.yaml
🚨

Nunca crie ReplicaSets diretamente em produção. Use Deployments — eles gerenciam os ReplicaSets por você, com rolling updates, rollbacks e histórico de revisões. O ReplicaSet criado manualmente não tem nenhum desses recursos.

DaemonSet: garantindo um Pod em cada Node

O DaemonSet é um objeto especial que garante que todos os Nodes do cluster executem exatamente uma cópia de um Pod específico. Quando um novo Node é adicionado ao cluster, um Pod é automaticamente criado nele. Quando um Node é removido, o Pod é removido junto.

O caso de uso mais claro é infraestrutura de observabilidade: você precisa que um agente de monitoramento (como o Node Exporter do Prometheus) rode em todos os Nodes para coletar métricas de CPU, memória e disco de cada um. Da mesma forma, um coletor de logs (Fluentd, Filebeat) precisa estar em cada Node para capturar logs de todos os containers.

Curiosidade: o próprio kube-proxy — que gerencia as regras iptables para os Services — é implementado como um DaemonSet no cluster!

Quando usar DaemonSet vs Deployment?

Use DaemonSet quando: a aplicação precisa rodar em cada Node (agentes de monitoramento, log collectors, drivers de armazenamento, plugins de rede). Use Deployment quando: você quer N réplicas de uma aplicação, não importa em quais Nodes estejam.

YAML · node-exporter-daemonset.yaml
apiVersion: apps/v1
kind: DaemonSet           # ← a única diferença visível do Deployment!
metadata:
  name: node-exporter
  labels:
    app: node-exporter
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      hostNetwork: true      # usa a rede do Node para coletar métricas reais
      containers:
      - name: node-exporter
        image: prom/node-exporter:latest
        ports:
        - containerPort: 9100
          hostPort: 9100      # expõe na porta do Node diretamente
        resources:
          limits:   {cpu: "200m", memory: "200Mi"}
          requests: {cpu: "100m", memory: "100Mi"}

Probes: ensinando o Kubernetes a saber quando sua app está saudável

Sem probes, o Kubernetes só sabe se o processo do container está rodando — ele não tem como saber se sua aplicação está realmente funcional. Imagine um servidor web que iniciou mas está travado em um deadlock: o processo existe, mas todas as requisições ficam penduradas para sempre. O K8s sem probe não saberia que algo está errado.

As probes resolvem isso. Existem três tipos, cada um com um propósito específico:

livenessProbe — "meu container ainda está vivo?"

A livenessProbe verifica continuamente se o container está funcionando corretamente. Se a probe falhar o número de vezes definido em failureThreshold, o kubelet mata e reinicia o container. Use para detectar situações em que a aplicação travou e não consegue se recuperar sozinha.

readinessProbe — "meu container está pronto para receber tráfego?"

A readinessProbe verifica se o container está pronto para receber requisições. Se falhar, o Pod é removido temporariamente do endpoint do Service — o tráfego para de chegar nele, mas o container não é reiniciado. Isso é essencial para zero-downtime: durante um rolling update, os novos Pods só começam a receber tráfego quando a readinessProbe passa.

startupProbe — "minha aplicação terminou de inicializar?"

A startupProbe é especialmente importante para aplicações lentas na inicialização (como aplicações Java com JVM, Spring Boot, etc.). Enquanto a startupProbe não passar, as outras probes ficam desabilitadas. Isso evita que o K8s reinicie desnecessariamente um container que simplesmente está demorando para carregar.

Três formas de configurar uma probe

  • httpGet: faz uma requisição HTTP. A probe passa se o status code for 200–399. Mais usado para APIs HTTP.
  • tcpSocket: tenta abrir uma conexão TCP na porta especificada. A probe passa se a conexão for estabelecida. Útil para bancos de dados, serviços que não são HTTP.
  • exec: executa um comando dentro do container. A probe passa se o código de saída for 0. Máxima flexibilidade.
YAML · deployment-com-probes.yaml (completo)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-com-probes
spec:
  replicas: 3
  selector:
    matchLabels: {app: nginx}
  template:
    metadata:
      labels: {app: nginx}
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        resources:
          limits:   {cpu: "0.5", memory: "256Mi"}
          requests: {cpu: "0.25", memory: "128Mi"}

        # startupProbe: aguarda até 150s (30 * 5s) para a app inicializar
        # Ideal para apps Java/Spring Boot que demoram para carregar
        startupProbe:
          httpGet:
            path: /
            port: 80
          failureThreshold: 30  # 30 tentativas × 5s = 150s máximo para startup
          periodSeconds: 5

        # livenessProbe: verifica se o container ainda está funcionando
        # Falha → container é reiniciado pelo kubelet
        livenessProbe:
          httpGet:
            path: /healthz      # endpoint de health da sua aplicação
            port: 80
          initialDelaySeconds: 10  # espera 10s antes da 1ª verificação
          periodSeconds: 10       # verifica a cada 10s
          failureThreshold: 3    # 3 falhas seguidas → reinicia o container
          timeoutSeconds: 5     # timeout de 5s por verificação

        # readinessProbe: verifica se o container está pronto para tráfego
        # Falha → Pod removido do Service (sem reinício)
        readinessProbe:
          httpGet:
            path: /ready
            port: 80
          initialDelaySeconds: 5
          periodSeconds: 5
          failureThreshold: 3
          successThreshold: 1   # 1 sucesso = volta ao Service
🚨

Armadilha clássica: configurar initialDelaySeconds: 0 com failureThreshold: 1 na livenessProbe vai fazer o K8s reiniciar seu container antes de ele ter chance de inicializar. O resultado é um CrashLoopBackOff que não é culpa da sua aplicação, mas da probe mal configurada. Para apps lentas, use sempre a startupProbe.

ℹ️

Analogia para fixar: pense na livenessProbe como o sinal de coração de um monitor hospitalar (se parar, o desfibrilador entra em ação). A readinessProbe é como a luz verde de "aberto" na porta de um restaurante (se apagar, novos clientes não entram, mas os que já estão dentro continuam sendo atendidos).

Preparando os servidores — cada passo importa

Esses passos devem ser executados em todos os nodes — tanto no control-plane quanto nos workers. Cada configuração tem uma razão técnica precisa; pular qualquer uma resultará em falhas silenciosas ou comportamentos imprevisíveis.

Por que desativar o swap?

O Kubernetes toma decisões de scheduling baseado em quanto de memória RAM está disponível em cada Node. Se o swap estiver ativo, um Pod pode parecer ter mais memória do que o Node realmente oferece em RAM física. O scheduler aloca o Pod baseado em uma premissa falsa, e o resultado pode ser desempenho degradado, OOM kills inesperados e comportamentos imprevisíveis. O kubeadm verificará e recusará a inicialização se o swap estiver ativo (a menos que você explicitamente ignore o aviso).

Por que carregar os módulos overlay e br_netfilter?

O módulo overlay é necessário para os sistemas de arquivos em camadas dos containers. O módulo br_netfilter permite que o netfilter (iptables) inspecione o tráfego que passa por bridges de rede — exatamente o que o kube-proxy precisa para rotear tráfego entre Pods em diferentes Nodes.

Day 6 — Volumes no Kubernetes
StorageClass · PersistentVolume · PersistentVolumeClaim · NFS · AWS EBS

Por que precisamos de volumes persistentes?

Containers, por design, são stateless — todo dado escrito dentro do container desaparece quando ele para. Se você rodar um MySQL dentro de um Pod sem nenhum volume persistente e o Pod for recriado (por qualquer motivo), todos os dados do banco de dados serão perdidos. É como reiniciar um servidor e perder o HD inteiro.

Para aplicações stateful — bancos de dados, sistemas de fila, aplicações que precisam gravar arquivos — você precisa de armazenamento que sobreviva à vida do Pod. É aí que entram os PersistentVolumes.

A hierarquia do armazenamento no K8s

O Kubernetes organiza o armazenamento em três camadas que trabalham juntas:

  1. StorageClass: descreve "classes" de armazenamento disponíveis no cluster (SSD rápido, HDD barato, NFS, etc.) e o provisioner responsável por criar volumes nessa classe.
  2. PersistentVolume (PV): representa um pedaço de armazenamento físico já provisionado — pode ser um disco na cloud, um diretório NFS, um volume EBS. É criado pelo admin do cluster ou automaticamente pela StorageClass.
  3. PersistentVolumeClaim (PVC): é o "pedido" da aplicação por armazenamento. A aplicação diz "preciso de 10Gi com leitura/escrita" e o K8s encontra (ou cria) um PV compatível e faz o binding.

StorageClass: definindo classes de armazenamento

A StorageClass é o objeto que permite o provisionamento dinâmico de volumes. Sem ela, um administrador precisaria criar manualmente cada PersistentVolume antes que qualquer aplicação pudesse usá-lo — inviável em clusters com centenas de aplicações.

Com StorageClass, o fluxo muda completamente: você define uma classe (ex: "gp2-ssd" usando discos EBS da AWS), e quando uma aplicação cria um PVC usando essa classe, o K8s automaticamente provisiona um volume EBS de tamanho e tipo corretos. Todo o gerenciamento manual desaparece.

Cada StorageClass tem um provisioner — o software responsável por criar o volume físico. Em clusters cloud, o provisioner integra com a API da cloud. Em ambientes on-premise, você pode usar NFS, Ceph, ou o provisioner local-path do Rancher.

YAML · storageclass.yaml

kind: StorageClass
metadata:
  name: giropops-sc

                          # Delete (padrão): exclui PV + storage quando PVC é deletado
                          # Recycle: deprecated, não use

# Immediate: provisiona imediatamente quando o PVC é criado
# WaitForFirstConsumer é preferível para evitar volumes em zonas erradas

A reclaimPolicy: Delete é conveniente mas perigosa: se alguém deletar o PVC por engano, o volume físico e todos os dados são permanentemente removidos. Em produção, use Retain e implemente um processo de limpeza manual controlado.

PersistentVolume e PersistentVolumeClaim: o contrato de armazenamento

O fluxo é simples na teoria mas poderoso na prática: o administrador ou a StorageClass cria um PV (o armazenamento real), e a aplicação cria um PVC (o pedido por armazenamento). O Kubernetes faz o binding — casa um PVC com um PV compatível levando em conta capacidade, accessModes e storageClassName.

Os AccessModes definem como o volume pode ser montado:

  • ReadWriteOnce (RWO): pode ser montado com leitura/escrita por um único Node. É o modo para bancos de dados single-instance como PostgreSQL, MySQL.
  • ReadOnlyMany (ROX): pode ser montado como somente leitura por múltiplos Nodes. Útil para dados estáticos compartilhados.
  • ReadWriteMany (RWX): pode ser montado com leitura/escrita por múltiplos Nodes simultantaneamente. Requer storage compatível — NFS, Ceph RBD, AWS EFS. Importante: discos EBS padrão da AWS não suportam RWX.
YAML · pv.yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: meu-pv
  labels:
    storage: local
spec:
  capacity:
    storage: 1Gi                          # capacidade total do volume
  accessModes:
    - ReadWriteOnce                       # RWO: um Node com R/W
  persistentVolumeReclaimPolicy: Retain  # não excluir automaticamente
  storageClassName: standard             # deve bater com o PVC
  hostPath:                              # tipo de storage: diretório no Node
    path: "/mnt/data"                    # use apenas para dev/test!
    # Em produção use: nfs, awsElasticBlockStore, gcePersistentDisk, etc.
YAML · pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: meu-pvc
spec:
  accessModes:
    - ReadWriteOnce             # deve bater com o accessMode do PV
  resources:
    requests:
      storage: 1Gi             # pede 1Gi (não pode exceder o PV)
  storageClassName: standard  # deve bater com a StorageClass e o PV

apiVersion: v1

ConfigMap: externalizando configurações da aplicação

Um dos princípios fundamentais de aplicações cloud-native (fator XII do 12-factor app) é separar configuração de código. Configurações mudam entre ambientes (dev, staging, prod) mas o código é o mesmo. O ConfigMap é a implementação do Kubernetes para esse princípio.

Um ConfigMap armazena dados não-sensíveis em pares chave-valor. Pode ser injetado nos Pods de três formas: como variáveis de ambiente, como argumentos de linha de comando ou como arquivos montados em volumes. Cada forma tem seu caso de uso ideal.

Como variável de ambiente é simples mas tem uma limitação importante: atualizar o ConfigMap não atualiza as variáveis em Pods já existentes — você precisa reiniciar os Pods. Como arquivo em volume tem uma vantagem: o Kubernetes propaga atualizações automaticamente em ~1 minuto, sem reiniciar o Pod. Ideal para arquivos de configuração que a aplicação consegue recarregar dinamicamente.

apiVersion: v1
kind: ConfigMap
metadata:

  DB_HOST: postgres.default.svc.cluster.local
  DB_PORT: "5432"

  MAX_CONNECTIONS: "100"

  # Arquivo completo — note o | que preserva as quebras de linha
  nginx.conf: |
    server {

      server_name _;

        proxy_pass http://backend:8080;

Secrets: gerenciando dados sensíveis

O Secret é estruturalmente muito similar ao ConfigMap — a diferença é a intenção: Secrets são para dados sensíveis como senhas, tokens de API, certificados TLS e chaves SSH. Eles recebem tratamento especial no Kubernetes: são armazenados no etcd separadamente, podem ter encryption at rest habilitada, e o RBAC pode restringir acesso a eles independentemente dos ConfigMaps.

Importante entender: por padrão, os valores em um Secret são apenas codificados em base64, que é encoding — não é criptografia. Qualquer pessoa com acesso ao etcd ou com permissão get secret no RBAC consegue decodificar o valor instantaneamente. Para segurança real, habilite Encryption at Rest no etcd e use ferramentas como HashiCorp Vault ou o External Secrets Operator para gerenciar credenciais de forma segura.

Tipos de Secrets

  • Opaque: tipo genérico — use para qualquer dado sensível.
  • kubernetes.io/tls: certificado e chave TLS. Usado por Ingress para HTTPS.
  • kubernetes.io/dockerconfigjson: credenciais de registry privado (para que o kubelet consiga fazer pull de imagens privadas).
  • kubernetes.io/service-account-token: token de ServiceAccount (gerado automaticamente pelo K8s).

StatefulSet: orquestrando aplicações com estado

O StatefulSet é o objeto ideal para gerenciar aplicações stateful — aquelas que precisam de identidade persistente, como bancos de dados relacionais (PostgreSQL, MySQL), sistemas de busca (Elasticsearch), filas de mensagens (Kafka, RabbitMQ) e caches distribuídos (Redis Cluster).

A diferença fundamental em relação ao Deployment é a identidade. Enquanto o Deployment trata todos os Pods como intercambiáveis e anônimos (seus nomes são gerados aleatoriamente como nginx-7d4c8f9b-x7k2p), o StatefulSet dá a cada Pod uma identidade estável e previsível: mysql-0, mysql-1, mysql-2. Se o Pod mysql-0 morrer e for recriado, ele volta com o mesmo nome, o mesmo PVC e, consequentemente, os mesmos dados.

As quatro garantias do StatefulSet

  1. Identidade de rede estável: cada Pod tem um nome fixo e um DNS único — pod-0.headless-svc.namespace.svc.cluster.local.
  2. Armazenamento persistente individual: cada Pod tem seu próprio PVC gerado automaticamente via volumeClaimTemplates. Deletar o Pod não deleta o PVC.
  3. Ordem de criação garantida: pod-0 é criado e deve estar Ready antes de pod-1 ser criado, e assim por diante.
  4. Ordem de remoção garantida: na remoção, os Pods são terminados do maior índice para o menor (pod-2 morre antes de pod-1, que morre antes de pod-0).

Esse comportamento é essencial para replicação master/replica em bancos de dados: mysql-0 é sempre o master, mysql-1 e mysql-2 são sempre as réplicas. A aplicação pode confiar nessas identidades.

apiVersion: apps/v1
kind: StatefulSet          # ← única diferença de kind vs Deployment
metadata:
  name: nginx
spec:
  serviceName: "nginx"     # OBRIGATÓRIO: nome do Headless Service associado
  replicas: 3
  selector:
    matchLabels: {app: nginx}
  template:
    metadata:
      labels: {app: nginx}
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
          name: web

          mountPath: "/usr/share/nginx/html"
  volumeClaimTemplates:     # ← cria um PVC INDIVIDUAL por Pod automaticamente!
  - metadata:
      name: www             # gera: www-nginx-0, www-nginx-1, www-nginx-2
    spec:
      accessModes: ["ReadWriteOnce"]
      storageClassName: standard
      resources:
        requests:
          storage: 1Gi

Headless Service: DNS individual por Pod

Todo StatefulSet precisa de um Headless Service associado. Mas o que é um Headless Service, e por que ele é necessário?

Um Service normal recebe um IP virtual (ClusterIP) e atua como load balancer — distribui o tráfego entre os Pods. Um Headless Service, definido com clusterIP: None, não tem IP próprio e não faz load balancing. Em vez disso, quando você consulta o DNS do Headless Service, recebe diretamente os IPs dos Pods individuais.

Para um StatefulSet, isso possibilita algo crítico: cada Pod recebe um endereço DNS individual e estável no formato:

<pod-name>.<service-name>.<namespace>.svc.cluster.local

Por exemplo: mysql-0.mysql.default.svc.cluster.local, mysql-1.mysql.default.svc.cluster.local. A réplica MySQL pode se conectar ao master usando exatamente esse endereço, com a certeza de que sempre chegará no Pod correto, independentemente de qual Node ele esteja rodando.

YAML · nginx-headless-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx        # deve bater com spec.serviceName do StatefulSet
  labels:
    app: nginx
spec:
  ports:
  - port: 80
    name: web
  clusterIP: None   # ← ISSO define o Headless Service
  selector:
    app: nginx      # seleciona os Pods do StatefulSet

Services: a abstração de rede que torna o K8s possível

Pods são efêmeros — quando recriados, ganham novos IPs. Se a sua aplicação frontend precisasse conhecer o IP do backend para se comunicar, seria um pesadelo: o IP muda a cada reinicialização. O Service resolve isso criando um endpoint de rede estável e permanente que aponta para um conjunto de Pods selecionados por labels.

Quando um Service é criado, o CoreDNS automaticamente cria uma entrada DNS para ele. O frontend pode simplesmente chamar http://backend-svc:8080 e o K8s cuida de resolver o nome e rotear para os Pods corretos — mesmo que os IPs mudem.

Endpoints: o que está por trás dos Services

Quando você cria um Service com um selector, o Kubernetes automaticamente cria um objeto Endpoints com o mesmo nome. Esse objeto lista os IPs e portas de todos os Pods que correspondem ao selector do Service. O kube-proxy lê esses Endpoints para configurar as regras iptables/IPVS que fazem o roteamento funcionar.

Inspecionar os Endpoints é uma ferramenta poderosa de debug: se um Service não está roteando tráfego corretamente, a primeira coisa a verificar é se os Endpoints estão populados. Se estiverem vazios, significa que nenhum Pod corresponde ao selector — provavelmente um erro de label.

BASH · INSPECIONANDO ENDPOINTS
# Ver Endpoints de um Service específico

# backend-svc   10.244.1.5:8080,10.244.2.3:8080   5m
#                 ↑ IPs dos Pods em execução

# Se Endpoints estiver vazio: labels não batem!
# NAME          ENDPOINTS   AGE
# backend-svc   <none>      5m  ← problema: nenhum Pod corresponde ao selector

# Debug: verificar os labels dos Pods
kubectl get pods --show-labels -l app=backend
# Se não retornar nada: o label 'app: backend' não existe nos Pods

# Verificar o selector do Service
kubectl describe service backend-svc | grep Selector
# Selector: app=backend

# Ver todos os Endpoints do namespace
kubectl get endpoints

# Uso avançado: Endpoint manual para serviço externo
# (Service sem selector + Endpoint manual = aponta para IP externo)
kubectl apply -f - <<EOF

kind: Service

  name: banco-externo
spec:
  ports:
  - port: 5432
---
apiVersion: v1
kind: Endpoints
metadata:
  name: banco-externo

- addresses:
  - ip: 192.168.1.100   # IP do banco de dados externo
  ports:
  - port: 5432
EOF

Endpoints manuais (sem selector) são úteis durante migrações: você pode migrar gradualmente do serviço externo para um Pod K8s apenas alterando o Endpoint, sem mudar como os outros serviços se conectam a ele. O nome do Service permanece o mesmo.

Ingress: roteamento HTTP/HTTPS de nível empresarial

O Ingress é o objeto responsável por gerenciar o acesso externo aos Services do cluster, principalmente para tráfego HTTP e HTTPS. Enquanto um Service do tipo LoadBalancer cria um IP público para cada Service (o que se torna caro e difícil de gerenciar em escala), o Ingress permite que você tenha um único ponto de entrada que roteia o tráfego para diferentes Services baseado no host ou no path da requisição.

É importante entender que o objeto Ingress, sozinho, não faz nada. Ele é apenas uma definição de regras. Para que essas regras sejam implementadas, você precisa de um Ingress Controller — um software que lê os objetos Ingress e configura um servidor de borda (geralmente nginx, Traefik ou HAProxy). O Ingress Controller não vem instalado por padrão no Kubernetes.

Ingress vs Service LoadBalancer

  • Service LoadBalancer: 1 LB por Service = custo alto, gerenciamento complexo. Adequado para alguns Services críticos.
  • Ingress: 1 LB (o do Ingress Controller) para N Services. Roteamento inteligente por host e path. TLS centralizado. Mais econômico e flexível.
BASH · INSTALANDO INGRESS CONTROLLER NGINX
# Instalar via Helm (recomendado)
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update
helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx --create-namespace

# Verificar que o controller está rodando
kubectl get pods -n ingress-nginx
kubectl get svc -n ingress-nginx
# ingress-nginx-controller   LoadBalancer   10.96.x.x   <EXTERNAL-IP>
YAML · ingress-completo.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    # Reescreve o path antes de passar para o Service
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx      # qual Ingress Controller usar
  tls:                          # TLS termination no Ingress
  - hosts: [meuapp.com]
    secretName: meuapp-tls     # Secret com cert.crt e cert.key
  rules:
  # Host-based routing: domínio diferente → Service diferente
  - host: meuapp.com
    http:
      paths:
      # Path-based routing: /api → backend, / → frontend
      - path: /api
        pathType: Prefix        # Prefix: /api, /api/users, /api/orders...
        backend:
          service:
            name: backend-svc
            port:
              number: 8080
      - path: /
        pathType: Prefix
        backend:
          service:
            name: frontend-svc
            port:
              number: 3000

Para TLS automático (Let\'s Encrypt), instale o cert-manager junto com o Ingress Controller. Com uma annotation (cert-manager.io/cluster-issuer: letsencrypt-prod), o cert-manager cuida automaticamente de provisionar e renovar os certificados TLS para cada domínio.

RBAC: quem pode fazer o quê no cluster

O RBAC (Role-Based Access Control) é o mecanismo de autorização do Kubernetes. Ele permite definir com precisão quem (usuário, grupo ou ServiceAccount) pode executar quais ações (get, list, create, delete, etc.) em quais recursos (pods, secrets, deployments...) em quais namespaces.

O RBAC tem quatro objetos principais que trabalham em dois pares:

  • Role + RoleBinding: permissões dentro de um namespace específico. Uma Role que dá permissão de listar pods no namespace "dev" não funciona no namespace "prod".
  • ClusterRole + ClusterRoleBinding: permissões no cluster inteiro, ou para recursos que não pertencem a namespaces (Nodes, PersistentVolumes, StorageClasses).

O fluxo é sempre: Role/ClusterRole define o que pode ser feito → RoleBinding/ClusterRoleBinding associa isso a um sujeito (User, Group ou ServiceAccount).

YAML · rbac-completo.yaml
# ── Role: permissões dentro de um namespace ───────────────────────
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: developer-role
  namespace: dev           # só vale neste namespace
rules:
- apiGroups: [""]          # "" = core API group (pods, services, etc.)
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]  # read-only
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "create", "update", "patch"]
---
# ── RoleBinding: associa Role a um usuário ────────────────────────
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developer-binding
  namespace: dev
subjects:
- kind: User               # User, Group ou ServiceAccount
  name: alice               # nome do usuário autenticado
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer-role
  apiGroup: rbac.authorization.k8s.io

# yes

Princípio do menor privilégio: nunca dê cluster-admin para uma aplicação. Crie ServiceAccounts específicas com apenas as permissões que a aplicação precisa. Uma aplicação comprometida com cluster-admin pode destruir o cluster inteiro.

HPA: Horizontal Pod Autoscaler

O HPA é o mecanismo do Kubernetes para escalar automaticamente o número de réplicas de um Deployment (ou StatefulSet) com base em métricas observadas — principalmente CPU e memória, mas também métricas customizadas via adaptadores.

O fluxo é: o HPA consulta o Metrics Server periodicamente (padrão: 15s). Se a métrica média dos Pods exceder o target, o HPA calcula o número ideal de réplicas usando a fórmula:

réplicas_desejadas = réplicas_atuais × (métrica_atual / métrica_target)

Por exemplo: 4 réplicas com CPU média de 80%, target de 50% → 4 × (80/50) = 6.4 → 7 réplicas.

O HPA tem um mecanismo de cooldown para evitar flapping (escalar e desescalar repetidamente): ele espera pelo menos 3 minutos antes de escalar para baixo novamente.

Pré-requisito obrigatório: o Metrics Server deve estar instalado no cluster. Sem ele, kubectl top pods não funciona e o HPA não consegue coletar métricas.

BASH · INSTALANDO METRICS SERVER
# Instalar via Helm
helm repo add metrics-server https://kubernetes-sigs.github.io/metrics-server/
helm install metrics-server metrics-server/metrics-server -n kube-system

# Para clusters locais (Kind/Minikube): adicionar flag de insecure TLS
helm install metrics-server metrics-server/metrics-server \
  --set args="{--kubelet-insecure-tls}" -n kube-system

# Testar se o Metrics Server está funcionando
kubectl top nodes
kubectl top pods
YAML · hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: meu-app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: meu-app
  minReplicas: 2     # nunca abaixo de 2 (alta disponibilidade)
  maxReplicas: 20    # nunca acima de 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60   # escala quando CPU média > 60% do request
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80   # escala quando memória média > 80%
BASH · TESTANDO O HPA

# NAME          REFERENCE       TARGETS    MINPODS  MAXPODS  REPLICAS

Ingress com Annotations: personalizando o comportamento do nginx

As Annotations do Ingress são a forma de passar configurações específicas ao Ingress Controller sem modificar o objeto padrão do Kubernetes. Cada controller tem suas próprias annotations — o nginx-ingress tem dezenas delas para controlar comportamentos como rewrite de URLs, rate limiting, autenticação, timeouts e muito mais.

As annotations mais usadas no nginx-ingress são:

  • rewrite-target: reescreve o path antes de passar ao Service. Útil quando seu Service espera / mas o Ingress roteia via /api.
  • ssl-redirect: redireciona automaticamente HTTP → HTTPS.
  • proxy-body-size: controla o tamanho máximo do body das requisições (padrão 1m).
  • proxy-read-timeout / proxy-send-timeout: timeouts de conexão com o backend.
  • use-regex: habilita regex nos paths do Ingress.
YAML · ingress-com-annotations.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: giropops-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/proxy-body-size: "8m"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    cert-manager.io/cluster-issuer: "letsencrypt-prod"  # TLS automático!
spec:
  ingressClassName: nginx
  tls:
  - hosts: [giropops.example.com]
    secretName: giropops-tls    # cert-manager preencherá este Secret
  rules:
  - host: giropops.example.com
    http:
      paths:
      - path: /senhas
        pathType: Prefix
        backend:
          service:
            name: giropops-senhas
            port: {number: 5000}

Cert-Manager: TLS automático com Let\'s Encrypt

O Cert-Manager é um controlador Kubernetes que automatiza todo o ciclo de vida dos certificados TLS: solicitação, emissão, renovação e rotação. Ele integra com diversas autoridades certificadoras, sendo o Let\'s Encrypt (gratuito) o mais usado.

O fluxo completo após a instalação: você cria um ClusterIssuer apontando para o Let\'s Encrypt → adiciona a annotation cert-manager.io/cluster-issuer no seu Ingress → o Cert-Manager detecta, solicita o certificado via desafio HTTP-01 ou DNS-01, recebe o certificado e o armazena em um Secret TLS. Quando o certificado está próximo de expirar (30 dias antes), o Cert-Manager renova automaticamente.

BASH + YAML · instalando e configurando Cert-Manager
# Instalar Cert-Manager via kubectl apply
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.14.1/cert-manager.yaml

# Confirmar instalação
kubectl get pods -n cert-manager
# cert-manager-xxx         Running
# cert-manager-webhook-xxx Running

# Criar ClusterIssuer para Let\'s Encrypt (válido para todo o cluster)
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: seuemail@exemplo.com          # seu email para notificações
    privateKeySecretRef:
      name: letsencrypt-prod-key
    solvers:
    - http01:                              # desafio HTTP (mais simples)
        ingress:
          class: nginx

# Verificar se certificado foi emitido
kubectl get certificate -A
kubectl describe certificate giropops-tls
💡

Use letsencrypt-staging durante testes (não tem limite de emissão). Só mude para letsencrypt-prod quando tudo estiver funcionando — o Let\'s Encrypt prod tem rate limit de 50 certificados por domínio por semana.

Labels e Annotations: a diferença que importa

Embora parecidos, Labels e Annotations têm propósitos completamente diferentes no Kubernetes.

Labels são pares chave-valor usados para identificação e seleção. Services, Deployments, ReplicaSets e NetworkPolicies usam labels para selecionar objetos. Eles são indexados pelo apiserver para queries eficientes. Devem ser concisos e seguir convenções. Exemplo: app: nginx, env: prod, version: v2.

Annotations são pares chave-valor para metadados não-identificadores. Não são usados para seleção — são para ferramentas externas, documentação, configuração de controllers. Podem ter valores grandes (JSON, YAML). Exemplo: configurações do Ingress Controller, causa de mudança no rollout, timestamp de último deployment.

BASH · gerenciando labels e annotations
# Adicionar label a um Pod existente
kubectl label pod meu-pod versao=v2

# Remover label (sufixo -)
kubectl label pod meu-pod versao-

# Selecionar recursos por label
kubectl get pods -l app=nginx
kubectl get pods -l "app=nginx,env=prod"
kubectl get pods -l "env in (prod,staging)"
kubectl get pods -l "env notin (dev)"

# Adicionar annotation
kubectl annotate deployment meu-app   kubernetes.io/change-cause="Deploy v2.1.0 - fix bug #123"

# Ver labels e annotations
kubectl get pods --show-labels
kubectl describe pod meu-pod | grep -A5 "Labels:\|Annotations:"

kube-prometheus-stack: observabilidade completa em um Helm Chart

O kube-prometheus-stack é o pacote Helm mais usado para monitoramento Kubernetes. Em um único helm install, ele instala: Prometheus (coleta e armazena métricas), Grafana (dashboards e visualização), AlertManager (roteamento de alertas para Slack, PagerDuty, etc.), Node Exporter (métricas de hardware dos Nodes), kube-state-metrics (métricas do estado dos objetos K8s) e Prometheus Operator (gerencia configs do Prometheus via CRDs).

O Prometheus Operator introduz dois conceitos importantes: ServiceMonitor e PodMonitor — CRDs que permitem configurar o scraping do Prometheus de forma declarativa, sem editar arquivos de configuração manualmente.

ServiceMonitor: configurando scraping declarativo

O ServiceMonitor é um CRD (Custom Resource Definition) do Prometheus Operator. Em vez de editar o arquivo prometheus.yml manualmente para adicionar novos endpoints de scraping, você cria um objeto ServiceMonitor que descreve quais Services o Prometheus deve monitorar. O Operator detecta o ServiceMonitor e reconfigura o Prometheus automaticamente.

Para que o Prometheus encontre o seu ServiceMonitor, o label do ServiceMonitor deve bater com o serviceMonitorSelector do Prometheus — que no kube-prometheus-stack usa release: monitoring por padrão.

YAML · servicemonitor.yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: minha-app-monitor
  labels:
    release: monitoring      # deve bater com serviceMonitorSelector do Prometheus
spec:
  selector:
    matchLabels:
      app: minha-app          # seleciona Services com este label
  endpoints:
  - port: metrics             # nome da porta no Service
    path: /metrics            # endpoint de métricas da aplicação
    interval: 30s             # frequência de scraping
  namespaceSelector:
    matchNames: [producao]   # namespace onde está o Service

Grafana: dashboards e alertas

O Grafana é a interface de visualização do stack. O kube-prometheus-stack já vem com dezenas de dashboards pré-configurados para o Kubernetes: uso de CPU/memória por namespace, status dos Pods, métricas dos Nodes, performance da rede, etc.

Para criar alertas que disparam para Slack ou PagerDuty quando algo está errado, você configura o AlertManager com receivers e routes, e cria PrometheusRules com as expressões PromQL que definem quando o alerta deve disparar.

YAML · prometheusrule.yaml (alerta de Pod CrashLoopBackOff)
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: alertas-kubernetes
  labels:
    release: monitoring
spec:
  groups:
  - name: kubernetes.rules
    rules:
    - alert: PodCrashLooping
      expr: rate(kube_pod_container_status_restarts_total[15m]) > 0
      for: 5m
      labels:
        severity: warning
      annotations:
        summary: "Pod {{ $labels.pod }} está em CrashLoopBackOff"
    - alert: NodeMemoryPressure
      expr: node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes < 0.1
      for: 2m
      labels:
        severity: critical
      annotations:
        summary: "Node {{ $labels.instance }} com menos de 10% de RAM livre"

Helm: o gerenciador de pacotes do Kubernetes

Imagine que você precisa instalar o PostgreSQL no seu cluster. Sem Helm, você teria que criar manualmente: StatefulSet, Service Headless, Service ClusterIP, ConfigMap com configurações, Secret com a senha, PersistentVolumeClaim, ServiceAccount, RBAC... São 7+ arquivos YAML. Com Helm: helm install meu-postgres bitnami/postgresql. Um comando.

O Helm trabalha com três conceitos centrais:

  • Chart: pacote reutilizável com templates dos manifestos Kubernetes + arquivo de valores padrão (values.yaml) + metadados (Chart.yaml). É o equivalente a um pacote npm ou pip.
  • Release: instância de um Chart instalado no cluster. Você pode instalar o mesmo Chart múltiplas vezes (postgres em diferentes namespaces) — cada instalação é uma Release independente com seu próprio nome e histórico.
  • Repository: coleção de Charts publicados. O Artifact Hub (artifacthub.io) é o repositório central de Charts públicos — como o npm registry para K8s.
BASH · comandos essenciais do Helm
# Adicionar repositório
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Buscar charts
helm search repo postgresql
helm search hub redis  # busca no Artifact Hub

# Instalar um chart
helm install meu-postgres bitnami/postgresql   --namespace databases   --create-namespace   --set auth.postgresPassword=senhasecreta

# Listar releases instaladas
helm list -A  # todos namespaces

# Ver valores padrão de um chart
helm show values bitnami/postgresql

# Upgrade de um release
helm upgrade meu-postgres bitnami/postgresql   --set replicaCount=2

# Histórico de revisões do release
helm history meu-postgres

# Rollback para revisão anterior
helm rollback meu-postgres 1

# Remover um release
helm uninstall meu-postgres -n databases

Criando seu próprio Helm Chart

Criar um Chart do zero é simples — o Helm gera a estrutura básica automaticamente. A partir daí, você substitui os manifests estáticos por templates com variáveis do values.yaml.

BASH · estrutura de um Chart
# Criar estrutura básica
helm create meu-app

# Estrutura gerada:
meu-app/
  Chart.yaml          # metadados: nome, versão, descrição
  values.yaml         # valores padrão (sobrescritos no install)
  templates/
    deployment.yaml   # template do Deployment
    service.yaml      # template do Service
    ingress.yaml      # template do Ingress
    hpa.yaml          # template do HPA
    _helpers.tpl      # funções reutilizáveis (naming conventions)
    NOTES.txt         # mensagem exibida após o install

# Renderizar templates localmente (sem instalar)
helm template meu-app ./meu-app

# Instalar localmente
helm install meu-app ./meu-app

# Com valores customizados por arquivo
helm install meu-app ./meu-app -f valores-producao.yaml

# Dry-run: valida sem instalar
helm install meu-app ./meu-app --dry-run --debug

Values e Templates: a mágica do Helm

Os templates do Helm usam a linguagem de templates do Go. Valores do values.yaml são referenciados como {{ .Values.campo }}. O Helm substitui essas expressões pelos valores reais durante a renderização.

  port: 80

  limits:   {cpu: "500m", memory: "256Mi"}

RBAC na prática: ServiceAccounts e permissões mínimas

O princípio de menor privilégio no Kubernetes significa que cada componente — seja um operador humano ou uma aplicação rodando em Pod — deve ter apenas as permissões estritamente necessárias. Uma aplicação comprometida com cluster-admin pode destruir todo o cluster; com permissões mínimas, o dano é contido.

Para aplicações, a identidade é a ServiceAccount. Por padrão, todo Pod usa a ServiceAccount default do namespace, que não tem permissões além de autenticar-se na API. Crie ServiceAccounts dedicadas por aplicação.

YAML · rbac-completo.yaml
# 1. ServiceAccount para a aplicação
apiVersion: v1
kind: ServiceAccount
metadata:
  name: minha-app-sa
  namespace: producao
---
# 2. Role com permissões mínimas necessárias
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: minha-app-role
  namespace: producao
rules:
- apiGroups: [""]
  resources: ["configmaps"]       # só lê ConfigMaps
  verbs: ["get", "list"]
- apiGroups: [""]
  resources: ["pods"]              # só lista pods
  verbs: ["get", "list", "watch"]
---
# 3. RoleBinding ligando ServiceAccount à Role
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: minha-app-binding
  namespace: producao
subjects:
- kind: ServiceAccount
  name: minha-app-sa
  namespace: producao
roleRef:
  kind: Role
  name: minha-app-role
  apiGroup: rbac.authorization.k8s.io
---
# 4. Deployment usando a ServiceAccount
spec:
  template:
    spec:
      serviceAccountName: minha-app-sa  # usa a SA criada
      automountServiceAccountToken: false  # segurança: não monta token se não precisar

Pod Security Standards: protegendo o cluster de Pods inseguros

O Pod Security Standards (PSS) é o mecanismo nativo do K8s (desde 1.25) para restringir o que um Pod pode fazer em termos de segurança. Existem três perfis:

  • Privileged: sem restrições — permite tudo, incluindo containers privilegiados. Usado apenas para infra de sistema (kube-system).
  • Baseline: restrições mínimas — impede escalada de privilégios óbvia mas permite a maioria das cargas de trabalho. Bom ponto de partida.
  • Restricted: restrições máximas — bloqueia tudo que não é estritamente necessário. Recomendado para produção.

O PSS é aplicado por namespace via labels. Existem três modos de aplicação: enforce (bloqueia), audit (registra no audit log), warn (exibe aviso mas não bloqueia).

BASH · aplicando Pod Security Standards
# Aplicar perfil restricted ao namespace producao (bloqueia + avisa)
kubectl label namespace producao   pod-security.kubernetes.io/enforce=restricted   pod-security.kubernetes.io/enforce-version=latest   pod-security.kubernetes.io/warn=restricted   pod-security.kubernetes.io/audit=restricted

# Tentar criar pod privilegiado será bloqueado:
# Error: pods "nginx" is forbidden: violates PodSecurity "restricted:latest"

# Ver labels de segurança de um namespace
kubectl get namespace producao --show-labels

Day 13 — Imagens Seguras com Wolfi e Distroless

Imagens Distroless: menos é mais seguro

Uma imagem Docker tradicional baseada em ubuntu:22.04 ou debian:bookworm carrega centenas de pacotes que sua aplicação nunca usa — bash, curl, apt, ferramentas de sistema... Cada pacote é uma superfície de ataque potencial com CVEs (Common Vulnerabilities and Exposures).

Imagens Distroless (criadas pelo Google) contêm apenas a sua aplicação e suas dependências de runtime — sem shell, sem gerenciador de pacotes, sem utilitários de sistema. O resultado são imagens menores, muito mais seguras e com pouquíssimos CVEs.

DOCKERFILE · multi-stage com imagem distroless
# Stage 1: BUILD (usa imagem completa com ferramentas de build)
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
RUN CGO_ENABLED=0 go build -o minha-app .

# Stage 2: RUNTIME (imagem distroless — sem shell!)
FROM gcr.io/distroless/static:nonroot
COPY --from=builder /app/minha-app /
USER 65532  # nonroot user
ENTRYPOINT ["/minha-app"]

# Resultado: imagem de ~5MB vs ~200MB com ubuntu base
# kubectl exec -it pod -- sh  ← FALHA: não tem shell (feature, not bug!)
💡

Para debugar containers distroless em produção, use o kubectl debug (disponível desde K8s 1.23) que adiciona um container temporário com ferramentas de debug sem modificar a imagem original: kubectl debug -it meu-pod --image=busybox --target=meu-container

Wolfi OS e Chainguard Images

O Wolfi é uma distribuição Linux minimalista criada pela Chainguard, projetada especificamente para containers seguros. Diferente das distroless (que não têm shell nem package manager), o Wolfi tem o apk mas é construído do zero com foco em segurança: zero CVEs conhecidos no momento da publicação, builds reproduzíveis e SBOM (Software Bill of Materials) para cada imagem.

As Chainguard Images são imagens públicas gratuitas baseadas em Wolfi para as stacks mais populares (nginx, node, python, go, java). São substitutas drop-in para as imagens oficiais do Docker Hub, mas com zero ou quase zero CVEs.

DOCKERFILE · usando Chainguard Images
# Antes: image com centenas de CVEs
FROM node:20-alpine

# Depois: Chainguard Node (zero CVEs conhecidos)
FROM cgr.dev/chainguard/node:latest

# Comparação de CVEs (exemplo real):
# node:20-alpine    → ~15 CVEs (HIGH/CRITICAL)
# chainguard/node   → 0 CVEs

Trivy: scanning de vulnerabilidades em imagens

O Trivy (da Aqua Security) é o scanner de vulnerabilidades open-source mais popular para containers. Ele analisa a imagem e reporta todos os CVEs conhecidos nos pacotes instalados, com severidade (CRITICAL, HIGH, MEDIUM, LOW).

BASH · Trivy na prática
# Instalar Trivy
curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b /usr/local/bin

# Scanear uma imagem
trivy image nginx:latest
# Total: 15 (CRITICAL: 2, HIGH: 5, MEDIUM: 8)

trivy image cgr.dev/chainguard/nginx:latest
# Total: 0 (CRITICAL: 0, HIGH: 0)

# Só mostrar CRITICAL e HIGH
trivy image --severity CRITICAL,HIGH minha-app:v1

# Integrar no CI/CD (falha o pipeline se houver CVEs críticos)
trivy image --exit-code 1 --severity CRITICAL minha-app:v1

Network Policies: o firewall dos Pods

Por padrão, no Kubernetes todos os Pods podem se comunicar com todos os outros Pods do cluster, em qualquer namespace. Isso é conveniente para desenvolvimento, mas é um pesadelo de segurança em produção: um Pod comprometido pode fazer requisições para qualquer outro serviço do cluster — incluindo o banco de dados de produção.

As Network Policies permitem restringir essa comunicação, implementando um modelo de firewall no nível de Pod. Elas funcionam como allow-lists: por padrão, tudo é permitido. Quando você cria uma NetworkPolicy que seleciona um Pod, ela implicitamente nega todo o tráfego não explicitamente permitido.

Pré-requisito crítico: as NetworkPolicies são apenas definições. Quem as implementa é o CNI plugin. Flannel não suporta NetworkPolicies. Use Calico, Cilium ou Weave Net.

Criando regras de NetworkPolicy

YAML · network-policies.yaml
# ── Default Deny ALL: bloqueia todo tráfego do namespace ──────────
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: producao
spec:
  podSelector: {}            # {} = seleciona TODOS os Pods
  policyTypes: [Ingress, Egress]  # bloqueia entrada e saída
---
# ── Permitir: frontend acessa backend na porta 8080 ───────────────
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: producao
spec:
  podSelector:
    matchLabels: {app: backend}   # regra se aplica ao backend
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels: {app: frontend}   # só aceita do frontend
    ports:
    - protocol: TCP
      port: 8080
---
# ── Permitir: DNS (CoreDNS) para todos ────────────────────────────
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: producao
spec:
  podSelector: {}
  policyTypes: [Egress]
  egress:
  - ports:
    - protocol: UDP
      port: 53   # DNS — sem isso, pods não resolvem nomes!
🚨

Ao aplicar um default-deny-all, lembre-se de criar explicitamente uma regra permitindo tráfego DNS (porta UDP 53) — sem ela, os Pods não conseguirão resolver nomes de Services e tudo quebrará silenciosamente.

Taints e Tolerations: Nodes que rejeitam Pods

Um Taint é uma marca em um Node que diz "não quero Pods aqui, a menos que eles explicitamente tolerem isso". Uma Toleration é a declaração do Pod dizendo "eu aceito ser colocado em Nodes com esse taint".

O caso de uso clássico é um Node com GPU: você adiciona o taint hardware=gpu:NoSchedule ao Node, e apenas Pods com a toleration correspondente serão agendados nele — evitando que Pods sem GPU desperdicem esse recurso caro.

Existem três efeitos possíveis para um Taint:

  • NoSchedule: novos Pods sem toleration não são agendados no Node. Pods existentes não são afetados.
  • PreferNoSchedule: o scheduler tenta evitar o Node, mas pode usá-lo se não houver alternativa.
  • NoExecute: Pods existentes sem toleration são evictados imediatamente, e novos não são agendados.
BASH + YAML · taints e tolerations na prática
# Adicionar taint a um Node
kubectl taint nodes gpu-node-01 hardware=gpu:NoSchedule

# Ver taints de um Node
kubectl describe node gpu-node-01 | grep Taints

# Remover taint (sufixo -)
kubectl taint nodes gpu-node-01 hardware=gpu:NoSchedule-

# Pod que tolera o taint e roda APENAS no Node com GPU
spec:
  tolerations:
  - key: "hardware"
    operator: "Equal"
    value: "gpu"
    effect: "NoSchedule"
  nodeSelector:
    hardware: gpu    # também força ir para Node com GPU (além de tolerar)

Node Affinity e Pod Affinity/Anti-Affinity

Enquanto Taints/Tolerations são a perspectiva do Node ("não aceito certos Pods"), o Affinity é a perspectiva do Pod ("prefiro ou exijo ir para certos Nodes"). O Affinity é mais expressivo que o nodeSelector, permitindo operadores como In, NotIn, Exists e regras required (hard) vs preferred (soft).

O Pod Anti-Affinity é especialmente poderoso para alta disponibilidade: você pode instruir o scheduler a nunca colocar duas réplicas do mesmo Deployment no mesmo Node ou mesmo AZ — garantindo que uma falha de Node/AZ não derrube todas as réplicas.

YAML · pod anti-affinity para HA
spec:
  template:
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:  # hard: obrigatório
          - labelSelector:
              matchLabels: {app: meu-app}        # evita Pods do mesmo app
            topologyKey: "kubernetes.io/hostname"  # no mesmo Node
        nodeAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:  # soft: preferência
          - weight: 100
            preference:
              matchExpressions:
              - key: disktype
                operator: In
                values: [ssd]   # prefere Nodes com SSD, mas não obrigatório

TopologySpreadConstraints: distribuição inteligente

O TopologySpreadConstraints é a versão moderna e mais flexível do Pod Anti-Affinity para distribuição entre zonas. Em vez de apenas "nunca no mesmo Node", você pode expressar "distribua as réplicas uniformemente entre as 3 zonas de disponibilidade, com no máximo 1 Pod de diferença entre elas".

YAML · distribuição entre AZs
spec:
  topologySpreadConstraints:
  - maxSkew: 1                        # máx de diferença entre zonas
    topologyKey: "topology.kubernetes.io/zone"  # distribui por AZ
    whenUnsatisfiable: DoNotSchedule  # bloqueia se não satisfeito
    labelSelector:
      matchLabels: {app: meu-app}
  - maxSkew: 1
    topologyKey: "kubernetes.io/hostname"  # e por Node dentro de cada AZ
    whenUnsatisfiable: ScheduleAnyway   # prefere, mas não obriga
    labelSelector:
      matchLabels: {app: meu-app}

Backup e Restore do etcd: a operação mais crítica

O etcd é o único componente stateful do control plane — é onde vive todo o estado do cluster. Fazer backup do etcd regularmente não é opcional em produção, é uma obrigação. Um cluster sem backup do etcd é um cluster à beira do precipício.

BASH · backup e restore do etcd
# ── BACKUP ────────────────────────────────────────────────────────
ETCDCTL_API=3 etcdctl snapshot save /backup/etcd-$(date +%Y%m%d-%H%M%S).db   --endpoints=https://127.0.0.1:2379   --cacert=/etc/kubernetes/pki/etcd/ca.crt   --cert=/etc/kubernetes/pki/etcd/server.crt   --key=/etc/kubernetes/pki/etcd/server.key

# Verificar integridade do backup
ETCDCTL_API=3 etcdctl snapshot status /backup/etcd-backup.db --write-out=table

# ── RESTORE (em caso de desastre) ─────────────────────────────────
# 1. Parar o kube-apiserver (remover do static pods)
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/

# 2. Restaurar o snapshot
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-backup.db   --data-dir=/var/lib/etcd-restore

# 3. Atualizar o path no manifesto do etcd
sudo sed -i 's|/var/lib/etcd|/var/lib/etcd-restore|g'   /etc/kubernetes/manifests/etcd.yaml

# 4. Restaurar o kube-apiserver
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

Upgrade seguro do cluster com kubeadm

Fazer upgrade do Kubernetes requer atenção e ordem precisa. O processo correto é: atualizar o control plane primeiro, depois os workers um a um. Nunca pule versões — o K8s suporta upgrade de apenas uma minor version por vez (1.27 → 1.28, não 1.27 → 1.29).

BASH · upgrade do cluster (1.28 → 1.29)
# ── NO CONTROL PLANE ──────────────────────────────────────────────
# 1. Remover hold e atualizar kubeadm
sudo apt-mark unhold kubeadm
sudo apt-get install -y kubeadm=1.29.0-00
sudo apt-mark hold kubeadm

# 2. Verificar o plano de upgrade
sudo kubeadm upgrade plan

# 3. Aplicar o upgrade do control plane
sudo kubeadm upgrade apply v1.29.0

# 4. Atualizar kubelet e kubectl no control plane
sudo apt-mark unhold kubelet kubectl
sudo apt-get install -y kubelet=1.29.0-00 kubectl=1.29.0-00
sudo apt-mark hold kubelet kubectl
sudo systemctl daemon-reload && sudo systemctl restart kubelet

# ── EM CADA WORKER (um de cada vez!) ──────────────────────────────
# 5. Drenar o Node (remove Pods graciosamente)
kubectl drain worker-01 --ignore-daemonsets --delete-emptydir-data

# 6. Atualizar kubeadm, kubelet, kubectl no worker
sudo apt-mark unhold kubeadm kubelet kubectl

GitOps com ArgoCD: o Git como fonte da verdade

GitOps é uma prática onde o repositório Git é a única fonte de verdade para o estado desejado do cluster. Ninguém aplica manifests manualmente com kubectl apply — uma ferramenta como o ArgoCD monitora o repositório Git e sincroniza automaticamente o cluster com o que está no repositório.

Benefícios: auditoria completa (quem mudou o quê quando via git log), rollback simples (git revert), revisão de mudanças via Pull Request, segurança (kubectl não fica nas mãos dos desenvolvedores).

BASH · instalando e configurando ArgoCD
# Instalar ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd -f   https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Acessar a UI
kubectl port-forward svc/argocd-server -n argocd 8080:443

# Obter senha inicial
kubectl -n argocd get secret argocd-initial-admin-secret   -o jsonpath="{.data.password}" | base64 -d

# Criar uma Application (sincroniza Git → Cluster)
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: meu-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/minha-org/k8s-manifests
    targetRevision: HEAD
    path: apps/meu-app
  destination:
    server: https://kubernetes.default.svc
    namespace: producao
  syncPolicy:
    automated:
      prune: true      # remove recursos deletados do Git
      selfHeal: true   # corrige mudanças manuais no cluster
